## Описание проекта

**UniDiagramAI** — это веб-приложение, интеллектуальный ассистент на базе мультиагентных LLM, который по текстовому описанию на естественном языке (русском или английском) создаёт диаграммы любых типов:

- UML (классов, последовательностей, прецедентов и др.)
- BPMN (бизнес-процессы)
- IDEF0
- Архитектурные схемы (C4, Archimate — в будущем)
- Маркетинговые воронки, CJM
- Строительные сетевые графики
- и любые другие визуальные модели

**Никакого кода, никакого drag-and-drop** — просто опишите, что нужно, и получите готовую диаграмму, которую можно тут же отредактировать в интерактивном графическом редакторе.

Проект разрабатывается в рамках конкурса ODS (март 2026) и нацелен на российский рынок, где востребованы инструменты с поддержкой русского языка и строгих нотаций.

## Проблема и целевая аудитория

### Кому нужно?

- **Бизнес-аналитики, системные аналитики, архитекторы ПО** — для описания процессов, архитектур, сценариев.
- **Проектировщики, методологи, специалисты по качеству** — для визуализации методик и стандартов.
- **Маркетологи, product-менеджеры** — для построения воронок продаж, карт пути клиента.
- **Инженеры, строители** — для сетевых графиков, IDEF0.
- **Все, кто регулярно создаёт визуальные модели** и хочет сэкономить время.

### Какая боль сегодня?

| Тип решения | Ограничения |
|-------------|-------------|
| **Ручные редакторы** (Visio, Draw.io, Miro) | Требуют часов на расстановку элементов, выравнивание, подбор нотации. Отвлекают от содержания. |
| **Специализированные инструменты** (Business Studio, Visual Paradigm) | Сложны, дороги, требуют длительного обучения. Избыточны для большинства задач. |
| **Текстовые генераторы** (PlantUML, Mermaid) | Требуют знания кода на специальных языках. Неудобны для аналитиков без технического бэкграунда. |
| **Существующие AI-помощники** | Либо ограничены одной нотацией, либо генерируют статичные картинки без возможности редактирования. Не понимают русский язык. |

**Результат:** низкая скорость документирования, высокий порог входа, разрозненные инструменты.

## Решение: что делает PoC

PoC (Minimum Viable Product) будет реализован за **4 недели** и продемонстрирует:

1. **Генерация трёх типов диаграмм**  
   - UML Sequence (диаграммы последовательности)  
   - BPMN (бизнес-процессы)  
   - Маркетинговая воронка (funnel)

2. **Чат-интерфейс** (правый сайдбар)  
   Пользователь вводит запрос на русском или английском.  
   Примеры:  
   *«Покажи последовательность оплаты заказа с проверкой платежа»*  
   *«Нарисуй воронку продаж: просмотр → корзина → оплата → доставка»*

3. **Мультиагентная обработка запроса**  
   - **Оркестратор** определяет тип диаграммы.  
   - **Агент уточнения** (при необходимости) задаёт уточняющие вопросы, если запрос неполный.  
   - **Агент структурирования** преобразует текст в JSON-описание элементов (валидируется Pydantic-схемой).  
   - **Агент валидации** проверяет логику (наличие начальных/конечных элементов, корректность связей).  
   - **Агент визуализации** рисует диаграмму с помощью специализированных библиотек:  
        - BPMN → `bpmn.io`  
        - UML Sequence → `jointjs` (кастомизированный рендерер)  
        - Воронка → самописный SVG-рендерер

4. **Интерактивный редактор** (левый сайдбар)  
   - Перетаскивание элементов, изменение их свойств (текст, цвет, тип)  
   - Сохранение изменений обратно в JSON и синхронизация с чатом  
   - Уточнение через чат после генерации: *«Сделай этот блок синим»*, *«Добавь шаг "проверка документа" после оплаты»*  
   - Экспорт в PNG, JPG, PDF

5. **Сквозное состояние (state/memory) диалога**  
   Система помнит предыдущие запросы и правки в рамках сессии, позволяя итеративно дорабатывать диаграмму.

## Архитектура

Мультиагентная архитектура обеспечивает надёжность, гибкость и высокое качество генерации.

<img width="697" height="346" alt="image" src="https://github.com/user-attachments/assets/0d75f992-fb80-431e-8df2-2bee36aba019" />

Архитектура UniDiagramAI построена как многоуровневая система с чётким разделением фронтенда, бэкенда и внешних сервисов. В основе лежит мультиагентный подход, где каждый агент выполняет строго определённую задачу, а управляет ими оркестратор.

Пользователь взаимодействует с веб-интерфейсом, написанным на React. Интерфейс разделён на две основные зоны: правый сайдбар — чат, куда вводятся запросы на русском или английском языке, и левый сайдбар — интерактивный графический редактор, построенный на библиотеках bpmn.io (для BPMN) и jointjs (для UML Sequence), а также самописном SVG-рендерере для маркетинговых воронок. Чат и редактор синхронизируются через общее состояние, которое хранится локально в браузере.

Когда пользователь отправляет запрос, он по WebSocket или REST попадает в бэкенд на FastAPI. Центральный элемент бэкенда — оркестратор, реализованный на Python с использованием LangGraph. Оркестратор управляет потоком обработки: сначала он может вызвать агента уточнения, который через LLM задаёт вопросы, если исходный запрос недостаточно полный. После уточнения или сразу передаёт запрос агенту структурирования. Этот агент использует LLM (OpenAI GPT‑4 или YandexGPT) и Pydantic‑схемы, чтобы преобразовать текст в строго типизированное JSON-описание диаграммы. Для повышения качества агент может обращаться к векторному хранилищу Qdrant, где хранятся few‑shot примеры.

Полученный JSON передаётся агенту валидации, который на чистом Python проверяет логику диаграммы: наличие начальных и конечных элементов, корректность связей, соблюдение правил выбранной нотации. Если валидация не проходит, агент отправляет обратно сообщение об ошибке, и оркестратор может повторно вызвать агента структурирования с уточнениями.

Когда JSON признан валидным, он передаётся агенту визуализации. Этот агент на основе JSON рендерит диаграмму с помощью соответствующих библиотек: bpmn.io для BPMN, jointjs для UML, самописного SVG‑движка для воронки. Результат в виде SVG возвращается в веб-интерфейс и отображается в графическом редакторе.

После этого пользователь может вручную перетаскивать элементы, менять их свойства, а также давать новые команды через чат (например, «сделай этот блок синим»). Оркестратор поддерживает состояние сессии, поэтому все правки — как ручные, так и текстовые — накапливаются в едином JSON‑представлении, которое синхронизируется между чатом и редактором. Такая двусторонняя связь позволяет сохранять ручные изменения при последующих уточнениях через AI.

Локальное хранилище браузера используется для сохранения текущего состояния и экспортированных диаграмм (PNG, JPG, PDF). Вся обработка данных (кроме вызовов LLM API) происходит локально, что обеспечивает приватность пользовательских данных.

## 🧩 Архитектура PoC
Ниже представлена схема мультиагентной архитектуры UniDiagramAI.

<img src="https://github.com/user-attachments/assets/2c9f1454-ce12-4148-a7c2-8539a963f254">

**Поток данных:**

1. Пользователь вводит запрос в чат.
2. Оркестратор получает запрос и при необходимости вызывает агента уточнения.
3. Агент структурирования формирует JSON-описание диаграммы.
4. Агент валидации проверяет корректность JSON и логику.
5. При ошибках — возврат на шаг 3 с указанием ошибки.
6. Агент визуализации рендерит диаграмму и передаёт её в редактор.
7. Пользователь редактирует диаграмму вручную или через чат.
8. Все изменения синхронизируются с JSON-состоянием.

## Технические преимущества

### 1. JSON против XML/кода
Использование промежуточного JSON-представления вместо прямой генерации XML-кода (например, BPMN 2.0 XML) позволяет:
- Снизить задержку генерации на ~43%
- Сократить количество выходных токенов более чем на 75%
- Упростить валидацию и отладку

### 2. Атомарное редактирование через Function Calling
Благодаря JSON, система может выполнять атомарные операции (добавление, удаление, перенаправление связей) через вызовы функций LLM, что на 50% надёжнее прямого манипулирования сложным XML-синтаксисом.

### 3. Двусторонняя связь (Bidirectional Linkage)
В отличие от инструментов, генерирующих статичные изображения, UniDiagramAI поддерживает связь между текстом и визуальными узлами. Это позволяет сохранять ручные правки пользователя при последующих итерациях уточнения диаграммы через AI-чат.

## Что НЕ входит в PoC (Out-of-scope)

- ❌ Поддержка всех видов UML (только Sequence)
- ❌ IDEF0 и сложные архитектурные нотации (C4, Archimate)
- ❌ Интеграция с внешними источниками данных (CRM, базы знаний, event logs)
- ❌ Многопользовательская работа и совместное редактирование
- ❌ Мобильная версия (только десктопный веб-интерфейс)
- ❌ Сохранение проектов в облаке (только локальное хранение в браузере)
- ❌ Поддержка других языков, кроме русского и английского

## Конкурентные преимущества

| Критерий | UniDiagramAI | Mermaid AI | DiagramGPT | Visio + Copilot |
|----------|--------------|------------|------------|-----------------|
| Тип диаграмм | Мультинотационный (BPMN, UML, воронки, IDEF0 — в планах) | Ограничен (блок-схемы, UML, mind map) | Ограничен (UML, блок-схемы) | Широкий |
| Способ ввода | Естественный язык (рус/англ) | Естественный язык → Mermaid | Естественный язык → код | Естественный язык |
| Редактирование | Интерактивный редактор | Нет (только код) | Базовый веб-редактор | Мощный редактор |
| Мультиагентность | Да (оркестратор + агенты) | Нет | Нет | Нет |
| Поддержка русского | **Да (основной фокус)** | Нет | Нет | Нет |
| Валидация логики | Да (агент валидации) | Нет | Нет | Нет |
| Цена | Freemium | Подписка | Подписка | Подписка M365 |

**Ключевые преимущества:**
- **Русский язык и локализация** — инструмент изначально оптимизирован для русскоязычной бизнес-терминологии и методологий.
- **Поддержка IDEF0** — редкая и востребованная функция для системных инженеров и госструктур.
- **Мультиагентная валидация** — гарантирует логическую корректность диаграмм, снижая риск ошибок.
- **Отсутствие промежуточного кода** — пользователь работает только с естественным языком.
